# Markdown 写法

### 标题与小标题

默认规则：标题不使用编号。

不推荐：

```
## 一. xxx
## 二. yyy
### 2.1 zzz
```

原因：

* 标题主要用于分块，而不是表达顺序
* 插入或调整结构时，编号容易失效并造成不一致

**允许的例外**
仅当满足以下条件时，才使用编号：

* 内容具有明确的**顺序依赖**（如步骤、流程、操作指引）
* 顺序不会频繁变动（结构稳定）

示例（允许）：

```
1. 安装
2. 运行
3. 输入参数
```

**限制**

* 标题层级（##、###）与编号体系不得混用表达同一层语义
* 若不确定是否需要编号，则不要编号

### 分割线（---）

默认规则：不使用分割线。

不推荐：

```
## XXX
内容

---

## YYY
```

原因：

* 标题本身已经提供了结构分隔
* 分割线通常不增加信息，仅增加视觉噪声

**允许的例外**
仅在以下情况使用：

* 需要分隔**不同语义类型的内容块**（例如：正文与附录、说明与示例）
* 不使用分割线会导致结构难以辨认

限制：

* 不得在连续标题之间使用分割线
* 不得重复使用分割线进行“视觉装饰”

### 加粗（**text**）

默认规则：谨慎使用加粗。

不推荐：

```
- **目的1**：解释1
- **目的2**：解释2
```

原因：

* 列表结构和标点应当/已经提供足够的层次信息
* 过多强调会降低整体可读性

**允许的例外**
仅在以下情况使用：

* 强调句子中的**关键信息或结论**
* 用于标记一小段**概括性短语**

示例（允许）：

```md
该函数的主要问题是**时间复杂度过高**。
```

限制：

* 不得在列表中对每一项重复加粗
* 不得将整句或大段文本加粗
* 若一段中多处需要加粗，应重新组织结构，而不是继续加粗


### 图表（Mermaid）

默认规则：在需要绘制图表是，可以使用 Mermaid。

**适用场景**

* 流程图
* 模块关系
* 状态转换
* 简单时序

**限制**

* 若文本已足够清晰，不强制使用图表
* 不为简单内容强行绘图
* Mermaid 图应保持简洁，避免过度细节

### 总体原则

* 优先保证结构清晰，而不是视觉“好看”
* 避免引入需要额外维护的格式（如编号、对齐）
* 若不确定某种格式是否必要，则不要使用
