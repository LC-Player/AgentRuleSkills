# 给大模型的代码要求

# 基本要求

### 代码注释规范

在生成或修改代码注释时，必须遵守以下原则：

#### 禁止“吵到眼睛的注释”

形如下面的注释，有时可起到分割代码，视觉强调的作用：
```python
#################
##### Start #####
#################
```
**但是**，在每个注释前用“======= 注释 =======”，“////// 注释 //////”等强调，通常是**没有**必要的。
这种注释会让用户以为“这里很重要”，反而增加阅读负担，可谓“吵到读者的眼睛”。
```c++
// bad

///////////// comment ///////////
void create() 
{
    ...
}

// good

// comment
void create() 
{
    ...
}

// or

/**
 * @brief ...
 * @param ...
 * @return ...
 */
void create() 
{
    ...
}

```

#### 禁止“翻译代码”的注释

- **不允许**写出仅逐行翻译代码字面含义的注释。
- **判断标准**：如果注释仅重复了代码本身已经表达的信息，而没有提供更高层次的意图、约束或用法说明，则属于“翻译代码”注释，严禁使用。

#### 鼓励“怎么用”的注释，“为什么这么写”的注释

- 面向**调用者**，说明：
  - 函数的正确调用方式、参数含义（特别是非显而易见的限制或副作用）
  - 返回值的结构或特殊取值（如 `null` 的含义）
  - 使用的前提条件（例如“必须先调用 `init()`”）
  - 典型的调用示例（简短即可）

- 面向**维护者**，解释**设计决策或非显而易见的逻辑原因**，包括：
  - 为什么选择某种算法或数据结构
  - 为什么要处理某个边界情况
  - 为什么看似冗余的检查是必要的
  - 为什么没有采用更简单的写法（比如受限于外部系统、性能瓶颈、历史兼容性）

#### 最好代码“自注释”，减少额外的注释
符号名本身应该解释自身的作用。如果代码质量好，简单易读，就不需要那么多注释。

#### 注释语言与风格

- 注释语言建议与项目中已有注释保持一致
- 对于公开 API（函数、类、模块）的注释，应使用目标语言的标准文档格式（如 Java 的 `/** */`、Python 的 docstring、TS 的 JSDoc），以支持 IDE 提示和文档生成工具。

#### 除非用户要求，否则不得删除原有注释

只要没有大幅度改动代码，必须保留原有注释（除非真的是很没有用的注释）。
包括在移动代码到新文件时，也必须将注释一并移动。
当然，如果用户要求清理注释，也应当满足用户的需求


#### 禁止在注释中复述设计原则、要求或无信息量的元评论
不要在代码注释中重复用户提示词中已经明确表达的意图，也不要添加只为“解释设计”而毫无信息量的空洞短语。
除非这种意图用户明确要求在代码中明确，或者是“不自明”的。
也就是不要“用户说一句，我重复一句”。


### 不确定时禁止臆测

- 如果用户需求、项目上下文或代码行为存在歧义，而该歧义你自身无法解决，会影响执行，你 **必须**：
  1. 明确指出哪些信息不明确；
  2. 提出具体的问题以获取必要澄清；
  3. 不得自行假设一个合理的解释并继续行动。

### 信息不足时，要求结构化澄清或强制引用来源

- 当用户问题缺少关键信息（例如没有给出文件路径、函数签名、错误日志、具体版本等），且你无法自动获取该信息，同时该信息会影响代码生成、分析或操作的正确性时，**不得**自行补全或使用模糊的常见值。**必须**提出**具体、可回答的问题** 索取缺失信息（例如：“请提供 `UserService` 类所在的文件路径”）。


### 代码生成时的“最小变更”与“完整性”规则

- 生成或修改代码时，应在确保代码逻辑一致，全局行为正确的情况下，优先 **局部修改**，避免重写整个文件，除非用户明确要求。
- **必须保证生成的代码语法完整、自洽**：
  - 不允许生成包含 `// TODO: finish this later` 而未提供任何实现框架，除非用户明确要求先提供占位符。
  - 不允许在代码块中省略关键部分并用 `...` 代替，除非用户已知该省略且仅需示意。

### 安全与敏感信息处理规则

- 除非用户要求，否则**不得**在输出中保留或生成明显的敏感信息示例，包括但不限于：
  - 真实 API 密钥、密码、JWT 示例值（使用 `your-api-key-here` 或 `xxxxxxxx` 代替）。
  - 硬编码的生产环境 IP、域名、数据库连接字符串。
- 如果用户无意间在代码或日志中粘贴了敏感信息，不应直接重复这些信息，而应提示用户“检测到可能的敏感信息，建议轮换/移除”，并只提供脱敏后的代码片段。


### 空指针处理规则：直接解引用，不检查不静默

规则：
- 如果某个指针（或引用）在当前逻辑路径上**预期永远不为空**，则**禁止**添加 `if (ptr == null)` 之类的空值检查。
- 对于Java，应当**直接解引用**该指针（调用其方法、访问其字段等）。
- 对于C++等解引用空指针时行为不定，应当断言指针不为空而非if。
- 如果因程序 bug 导致指针实际为空，运行时将立刻抛出空指针异常（如 Java 的 `NullPointerException`、C++ 的 segmentation fault或断言失败），这比静默返回、吞掉错误或执行替代逻辑更安全。

原理：
- 编译时错误 > 运行时异常 > 静默错误 / 错误恢复
- 过早的防御性检查会掩盖真实的 bug，导致问题延迟暴露、难以定位
- 快速失败（fail-fast）原则：让错误在最近的位置、最明显的方式暴露出来

**反例**（不要这样写）
```java
// 逻辑上 user 绝对不应为 null
if (user != null) {
    user.updateProfile(...);
}
```

**正例**（应该这样写）
```java
// 直接调用，若 user 意外为 null 则程序立即崩溃并输出堆栈
user.updateProfile(...);
```

**例外情况**
- 只有当指针在**业务上允许为空**，且你需要显式处理空值逻辑时，才可以使用 `if (ptr == null)`（例如返回默认值、抛出业务异常）。
- 与外部输入（API 响应、用户输入）交互时，应在边界处校验并转换为非空域，内部则遵守本规则。

**备注**
这一条规则依赖于代码中充分利用空指针，减少野指针保证安全性，例如诸如C++的内存释放之后立即将指针赋空。这也是一条重要的原则。
这一条规则仅适用于“逻辑上指针不得为空”的情况。如果业务上指针可以为空，应当处理空指针情况

---

- 本规则适用于所有项目和编程语言。
- 若用户在当前对话中显式给出的指示与本规则冲突，以**用户当前指示**为准。但 你 应提示用户存在冲突，并确认用户的意图是否覆盖本规则。
- 若项目自身文档（如 `CONTRIBUTING.md`、`README.md`）中存在与本规则相悖的约定，你 应优先遵循项目约定，但需向用户说明差异。

# 代码对齐规则

0. 此处所指代码对齐，不是指缩进对齐，而是指额外空格的对齐

## 默认规则

   * 虽然有时在代码中添加空格进行列对齐在视觉上有一定作用，但由于维护困难，应当谨慎添加额外的空格强行对齐。
   * 通常行尾注释与代码之间，代码的token和token之间最多保留一个空格。
   * 不为了“视觉整齐”在普通语句中插入多余空格。
不好的例子：
        x = self.pool(F.relu(self.conv1(x)))   # -> B x A x H1 x W1
        x = self.pool(F.relu(self.conv2(x)))   # -> B x B x H2 x W2
        x = x.view(x.size(0), -1)              # 展平: B x (B*H2*W2)
        x = self.fc1(x)                        # -> B x 25
后两行为了对齐上文，代码和注释间用了数不清的空格，这是没有必要的。
推荐：

   ```python
   x = self.pool(F.relu(self.conv1(x))) # -> B x C1 x H1 x W1
   x = self.pool(F.relu(self.conv2(x))) # -> B x C2 x H2 x W2
   x = x.view(x.size(0), -1) # 展平: B x (C2*H2*W2)
   x = self.fc1(x) # -> B x 25
   ```

## 允许的例外
   在以下情况，可以使用对齐：

   * 多行**结构完全一致**（如 switch/case、大量变量初始化、表驱动映射、配置列表）
   * 右侧表达式形态一致（例如都是构造对象或简单赋值）
   * 代码很可能被**批量修改**（例如统一重命名变量）

   示例（允许）：

   ```cpp
   case NoteType::TAP:   sprite = new TapNoteSprite();   break;
   case NoteType::HOLD:  sprite = new HoldNoteSprite();  break;
   case NoteType::TOUCH: sprite = new TouchNoteSprite(); break;
   ```

## 限制范围（防止滥用）

   * 对齐只允许出现在**局部连续代码块**中，不得扩散到整个函数或文件
   * 不允许跨不同语句类型对齐（例如把声明、赋值、调用混在一起对齐）
   * 一旦结构不再一致，应立即停止对齐

## 优先级原则

   * 可读性优先于对齐
   * 简洁性优先于整齐
   * 若不确定是否应对齐，则不要对齐

> 以上内容仅针对需要额外空格进行对齐的情况，代码缩进依然按照最佳实践和规范进行书写。

# 代码风格

代码风格应当：
1. 首先按照用户的说明为准
2. 其次按照文档中的描述（如果有）
3. 最后参照上下文的代码风格
4. 如果无法判断，则按照当前语言的最佳实践为准。

# Markdown 写法

### 标题与小标题

默认规则：标题不使用编号。

不推荐：

```
## 一. xxx
## 二. yyy
### 2.1 zzz
```

原因：

* 标题主要用于分块，而不是表达顺序
* 插入或调整结构时，编号容易失效并造成不一致

**允许的例外**
仅当满足以下条件时，才使用编号：

* 内容具有明确的**顺序依赖**（如步骤、流程、操作指引）
* 顺序不会频繁变动（结构稳定）

示例（允许）：

```
1. 安装
2. 运行
3. 输入参数
```

限制：

* 标题层级（##、###）与编号体系不得混用表达同一层语义
* 若不确定是否需要编号，则不要编号

### 分割线（---）

默认规则：不使用分割线。

不推荐：

```
## XXX
内容

---

## YYY
```

原因：

* 标题本身已经提供了结构分隔
* 分割线通常不增加信息，仅增加视觉噪声

仅在以下情况使用：

* 需要分隔**不同语义类型的内容块**（例如：正文与附录、说明与示例）
* 不使用分割线会导致结构难以辨认

限制：

* 不得在连续标题之间使用分割线
* 不得重复使用分割线进行“视觉装饰”

### 加粗（**text**）

默认规则：谨慎使用加粗。

不推荐：

```
- **目的1**：解释1
- **目的2**：解释2
```

原因：

* 列表结构和标点应当/已经提供足够的层次信息
* 过多强调会降低整体可读性

仅在以下情况使用：

* 强调句子中的**关键信息或结论**
* 用于标记一小段**概括性短语**

示例（允许）：

```md
该函数的主要问题是**时间复杂度过高**。
```

限制：

* 不得在列表中对每一项重复加粗
* 不得将整句或大段文本加粗
* 若一段中多处需要加粗，应重新组织结构，而不是继续加粗


### 图表（Mermaid）

默认规则：在需要绘制图表是，可以使用 Mermaid。

适用场景

* 流程图
* 模块关系
* 状态转换
* 简单时序

限制

* 若文本已足够清晰，不强制使用图表
* 不为简单内容强行绘图
* Mermaid 图应保持简洁，避免过度细节

### 总体原则

* 优先保证结构清晰，而不是视觉“好看”
* 避免引入需要额外维护的格式（如编号、对齐）
* 若不确定某种格式是否必要，则不要使用

# 规则优先级

如果用户对话中有明确要求，以用户为准。

# 编程语言特定规则

- Python：
应当尽可能进行类型注解。
遵循这种思考原则：“如果在传统的严格类型语言（C++、Java）中，同样的设计必须完全抛弃静态类型检查和编译时约束，而靠大量的运行时类型判断和动态特性来凑效，那么这种设计在 Python 里也是不好的、难以维护的，应避免在大型项目中使用。”

- C++：
只要不影响行为，头文件和源文件的第一行，使用注释标出文件名。例如：
```c++
// main.cpp

#include <iostream>
...
```
