### 默认不执行任何动态操作

- **除非用户在当前对话中明确要求**（例如使用“请编译”、“运行测试”、“执行构建”、“启动程序”等显式指令），否则 Agent **不得**主动进行任何形式的：
  - 编译
  - 构建
  - 测试
  - 运行/执行
  - 部署
  - 安装依赖
  - 修改环境配置
  - 或其他会改变项目状态、产生副作用、需要实际计算资源或耗时等待的操作。

- **适用说明**：该规则对解释型语言（如 Python、JavaScript）、编译型语言（如 C++、Rust）、混合型项目（如 Java、Go）以及数据科学脚本、配置文件等均同等适用。“运行”包括但不限于执行主程序、调用解释器、启动容器、运行单元测试等。

- **例外处理**：如果用户要求中带有明确的条件或限定（例如“如果……则编译”），Agent 必须：
  1. 如实判断条件是否满足；
  2. 若条件模糊或无法从上下文中确定，**必须停止并询问用户**，不得自行臆测条件为真。

### 代码注释规范

Agent 在生成或修改代码注释时，必须遵守以下原则：

#### 禁止“翻译代码”的注释

- **不允许**写出仅逐行翻译代码字面含义的注释。
- **判断标准**：如果注释仅重复了代码本身已经表达的信息，而没有提供更高层次的意图、约束或用法说明，则属于“翻译代码”注释，严禁使用。

#### 鼓励“怎么用”的注释，“为什么这么写”的注释

- 面向**调用者**，说明：
  - 函数的正确调用方式、参数含义（特别是非显而易见的限制或副作用）
  - 返回值的结构或特殊取值（如 `null` 的含义）
  - 使用的前提条件（例如“必须先调用 `init()`”）
  - 典型的调用示例（简短即可）

- 面向**维护者**，解释**设计决策或非显而易见的逻辑原因**，包括：
  - 为什么选择某种算法或数据结构
  - 为什么要处理某个边界情况
  - 为什么看似冗余的检查是必要的
  - 为什么没有采用更简单的写法（比如受限于外部系统、性能瓶颈、历史兼容性）

#### 最好代码“自注释”，减少额外的注释
符号名本身应该解释自身的作用。如果代码质量好，简单易读，就不需要那么多注释。

#### 除非用户要求，否则不得删除原有注释

只要没有大幅度改动代码，必须保留原有注释（除非真的是很没有用的注释）。
包括在移动代码到新文件时，也必须将注释一并移动。
当然，如果用户要求清理注释，也应当满足用户的需求

#### 禁止在注释中复述设计原则、要求或无信息量的元评论
不要在代码注释中重复用户提示词中已经明确表达的意图，也不要添加只为“解释设计”而毫无信息量的空洞短语。
除非这种意图用户明确要求在代码中明确，或者是“不自明”的。
也就是不要“用户说一句，我重复一句”。

#### 注释语言与风格

- 注释语言建议与项目中已有注释保持一致（如果项目无既定风格，默认使用**英文**或**中文**均可，但需全文统一）。
- 对于公开 API（函数、类、模块）的注释，应使用目标语言的标准文档格式（如 Java 的 `/** */`、Python 的 docstring、TS 的 JSDoc），以支持 IDE 提示和文档生成工具。

### 不确定时禁止臆测

- 如果用户需求、项目上下文或代码行为存在歧义，而该歧义Agent自身无法解决，会影响执行，Agent **必须**：
  1. 明确指出哪些信息不明确；
  2. 提出具体的问题以获取必要澄清；
  3. 不得自行假设一个合理的解释并继续行动。

### 信息不足时，要求结构化澄清或强制引用来源

- 当用户问题缺少关键信息（例如没有给出文件路径、函数签名、错误日志、具体版本等），且Agent无法自动获取该信息，同时该信息会影响代码生成、分析或操作的正确性时，Agent **不得**自行补全或使用模糊的常见值。**必须**提出**具体、可回答的问题** 索取缺失信息（例如：“请提供 `UserService` 类所在的文件路径”）。

### 依赖、库、工具版本的规则
- **严禁**修改 `package.json`、`requirements.txt`、`go.mod`、`Cargo.toml` 等锁文件中的版本约束。


### 代码生成时的“最小变更”与“完整性”规则

- Agent 生成或修改代码时，应在确保代码逻辑一致，全局行为正确的情况下，优先 **局部修改**，避免重写整个文件，除非用户明确要求。
- **必须保证生成的代码语法完整、自洽**：
  - 不允许生成包含 `// TODO: finish this later` 而未提供任何实现框架，除非用户明确要求先提供占位符。
  - 不允许在代码块中省略关键部分并用 `...` 代替，除非用户已知该省略且仅需示意。

### 安全与敏感信息处理规则

- Agent **不得**在输出中保留或生成明显的敏感信息示例，包括但不限于：
  - 真实 API 密钥、密码、JWT 示例值（使用 `your-api-key-here` 或 `xxxxxxxx` 代替）。
  - 硬编码的生产环境 IP、域名、数据库连接字符串。
- 如果用户无意间在代码或日志中粘贴了敏感信息，Agent 不应直接重复这些信息，而应提示用户“检测到可能的敏感信息，建议轮换/移除”，并只提供脱敏后的代码片段。

### 空指针处理规则：直接解引用，不检查不静默

规则：
- 如果某个指针（或引用）在当前逻辑路径上**预期永远不为空**，则**禁止**添加 `if (ptr == null)` 之类的空值检查。
- 对于Java，应当**直接解引用**该指针（调用其方法、访问其字段等）。
- 对于C++等解引用空指针时行为不定，应当断言指针不为空而非if。
- 如果因程序 bug 导致指针实际为空，运行时将立刻抛出空指针异常（如 Java 的 `NullPointerException`、C++ 的 segmentation fault或断言失败），这比静默返回、吞掉错误或执行替代逻辑更安全。

原理：
- 编译时错误 > 运行时异常 > 静默错误 / 错误恢复
- 过早的防御性检查会掩盖真实的 bug，导致问题延迟暴露、难以定位
- 快速失败（fail-fast）原则：让错误在最近的位置、最明显的方式暴露出来

**反例**（不要这样写）
```java
// 逻辑上 user 绝对不应为 null
if (user != null) {
    user.updateProfile(...);
}
```

**正例**（应该这样写）
```java
// 直接调用，若 user 意外为 null 则程序立即崩溃并输出堆栈
user.updateProfile(...);
```

**例外情况**
- 只有当指针在**业务上允许为空**，且你需要显式处理空值逻辑时，才可以使用 `if (ptr == null)`（例如返回默认值、抛出业务异常）。
- 与外部输入（API 响应、用户输入）交互时，应在边界处校验并转换为非空域，内部则遵守本规则。

**备注**
这一条规则依赖于代码中充分利用空指针，减少野指针保证安全性，例如诸如C++的内存释放之后立即将指针赋空。这也是一条重要的原则。
这一条规则仅适用于“逻辑上指针不得为空”的情况。如果业务上指针可以为空，应当处理空指针情况

- 若用户在当前对话中显式给出的指示与本规则冲突，以**用户当前指示**为准。但 Agent 应提示用户存在冲突，并确认用户的意图是否覆盖本规则。
- 若项目自身文档（如 `CONTRIBUTING.md`、`README.md`）中存在与本规则相悖的约定，Agent 应优先遵循项目约定，但需向用户说明差异。
